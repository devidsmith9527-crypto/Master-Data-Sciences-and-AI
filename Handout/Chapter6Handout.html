១. សេចក្តីផ្តើមអំពី Graphs (រចនាសម្ព័ន្ធបណ្តាញតភ្ជាប់)
Graph (ក្រាហ្វ ឬបណ្តាញតភ្ជាប់) គឺជាប្រភេទរចនាសម្ព័ន្ធទិន្នន័យមិនលីនេអ៊ែរ (Non-linear Data Structure) ដ៏មានឥទ្ធិពលបំផុត ដែលត្រូវបានរចនាឡើងដើម្បីតំណាងឱ្យទំនាក់ទំនងពហុទិសដៅ **(Many-to-Many Relationship)**។ ខុសប្លែកពី Tree ដែលទិន្នន័យត្រូវតម្រៀបតាមឋានានុក្រមពីលើចុះក្រោម និងមាន Root តែមួយគត់ Graph មិនមានចំណុច Root ឬច្បាប់បិទបាំងអ្វីឡើយ។ ធាតុនីមួយៗអាចតភ្ជាប់ទៅកាន់ធាតុដទៃទៀតបានដោយសេរី បង្កើតបានជាបណ្តាញខ្វាត់ខ្វែង។
ករណីសិក្សាទី១៖ ប្រព័ន្ធផែនទី Google Maps

នៅក្នុងប្រព័ន្ធផែនទីរបស់ Google ទីក្រុង ឬចំណុចបំបែកនីមួយៗត្រូវបានតំណាងដោយចំណុច **Vertex (Node)**។ រីឯផ្លូវថ្នល់ ផ្លូវល្បឿនលឿន ឬស្ពានដែលតភ្ជាប់ពីក្រុងមួយទៅក្រុងមួយទៀត ត្រូវបានតំណាងដោយខ្សែ **Edge**។ Google ប្រើរចនាសម្ព័ន្ធនេះដើម្បីគណនារកផ្លូវខ្លីបំផុត O(V + E)។

២. សទ្ទានុក្រមស្នូលរបស់ Graphs (Vertices & Edges)
ដើម្បីសរសេរកូដគ្រប់គ្រង និងអភិវឌ្ឍប្រព័ន្ធ AI ដែលពឹងផ្អែកលើ Graph យើងត្រូវយល់ច្បាស់ពីលក្ខណៈបច្ចេកទេសស្នូលទាំងនេះ៖

១. Vertex (ឬ Node - ចំណុចគោល)

ជាប្រអប់ ឬវត្ថុដែលរក្សាទុកទិន្នន័យពិតប្រាកដ។ ឧទាហរណ៍៖ នៅក្នុងបណ្តាញអ៊ីនធឺណិត Vertex គឺជារ៉ោតទ័រ (Router) ឬកុំព្យូទ័ររបស់អតិថិជន។

២. Edge (ខ្សែតភ្ជាប់ទំនាក់ទំនង)

ជាខ្សែចងភ្ជាប់ទំនាក់ទំនងរវាង Vertex ពីរ។ អាចមានលក្ខណៈគ្មានទិសដៅ **(Undirected Edge)** ដូចជាបណ្តាញទំនាក់ទំនងមិត្តភាព ឬមានទិសដៅជាក់លាក់ **(Directed Edge)** ដូចជាបណ្តាញ Follower របស់ Instagram (A តាមដាន B តែ B មិនប្រាកដថាផ្ញើត្រឡប់មក A វិញឡើយ)។

៣. Weight (ទម្ងន់ខ្សែ ឬតម្លៃចំណាយ)

ជាតម្លៃលេខដែលបានកំណត់ដាក់នៅលើ Edge នីមួយៗ ដើម្បីតំណាងឱ្យចម្ងាយ (គីឡូម៉ែត្រ) រយៈពេល (នាទី) ឬតម្លៃប្រើប្រាស់ផ្សេងៗ។ ក្រាហ្វដែលមាន Weight ហៅថា **Weighted Graph**។

៤. Cycle (រង្វិលជុំក្នុងបណ្តាញ)

ជាគន្លងផ្លូវ (Path) ដែលចាប់ផ្តើមចេញពី Vertex មួយ ហើយបន្តឆ្លងកាត់ខ្សែតភ្ជាប់ Edges ផ្សេងៗរួចវិលមកជាន់ចំណុច Vertex ដើមវិញជាលក្ខណៈវង្វង់បិទជិត។

ករណីសិក្សា៖ ស្ទះចរាចរណ៍ និង Cycle Detected

នៅក្នុងប្រព័ន្ធចរាចរណ៍ក្រុងឆ្លាតវៃ (Smart City AI) ម៉ាស៊ីនវិភាគនឹងចាត់ទុកផ្លូវថ្នល់ជា Directed Graph។ ប្រសិនបើ AI ឆែករកឃើញថាមានការកកើតឡើងនៃ **Cycle (រង្វិលជុំជាប់គាំង)** នៅក្នុង Graph នោះមានន័យថាចរាចរណ៍កំពុងជួបបញ្ហាស្ទះរង្វង់មូល ដែលទាមទារឲ្យមានការប្តូរភ្លើងសញ្ញាចរាចរណ៍ជាបន្ទាន់។

៣. យន្តការតំណាង Graph តាមរយៈ Adjacency List
នៅក្នុងការសរសេរកូដ តើកុំព្យូទ័ររក្សាទុក Graph យ៉ាងដូចម្តេច? វិធីពេញនិយម និងមានប្រសិទ្ធភាពបំផុតគឺប្រើប្រាស់ Adjacency List (បញ្ជីអ្នកជិតខាង)។ យន្តការនេះប្រើប្រាស់ Python Dictionary ដើម្បីផ្ទុក Vertex នីមួយៗជាកូនសោ (Key) ហើយ Value របស់វាគឺជាបញ្ជី (List) ផ្ទុកនូវរាល់គ្រប់ចំណុចដែលបានភ្ជាប់មកកាន់វាផ្ទាល់។

class Graph:
    def __init__(self):
        # បង្កើត Dictionary ទទេមួយសម្រាប់ផ្ទុក Adjacency List
        self.adj_list = {}

    # បន្ថែមគែមតភ្ជាប់រវាង Vertex u និង v O(1)
    def add_edge(self, u, v):
        # ប្រសិនបើចំណុច u ឬ v មិនទាន់មានក្នុងបញ្ជី ត្រូវបង្កើត List ទទេមួយជាមុនសិន
        if u not in self.adj_list:
            self.adj_list[u] = []
        if v not in self.adj_list:
            self.adj_list[v] = []

        # ភ្ជាប់ u ទៅ v ហើយនិងភ្ជាប់ v ទៅ u វិញ (Undirected Graph)
        self.adj_list[u].append(v)
        self.adj_list[v].append(u)

    # បង្ហាញបណ្តាញតភ្ជាប់សរុប O(V)
    def display_graph(self):
        for vertex, neighbors in self.adj_list.items():
            print(f"{vertex} តភ្ជាប់ទៅកាន់ -> {neighbors}")

# ដំណើរការសាកល្បង
network = Graph()
network.add_edge("A", "B")
network.add_edge("B", "C")
network.add_edge("A", "D")
network.add_edge("B", "D")

print("រចនាសម្ព័ន្ធបណ្តាញតភ្ជាប់ Graph គំរូ:")
network.display_graph()

៥. ក្បួនដោះស្រាយរុករកក្នុង Graph (BFS និង DFS)
ដោយសារ Graph អាចតភ្ជាប់ខ្វាត់ខ្វែងគ្មានទីបញ្ចប់ ដើម្បីរុករកបង្ហាញ ឬស្វែងរកទិន្នន័យ វិស្វករ DSA បានបង្កើតក្បួនដោះស្រាយស្វែងរកធំៗពីរ គឺ BFS និង DFS៖

១. BFS (Breadth-First Search - ស្វែងរកតាមទម្រង់វាលរាប)

BFS ធ្វើការរុករកដោយស្កែនមើលរាល់គ្រប់ចំណុចអ្នកជិតខាងដែលនៅជិតបង្អស់ជាមុនសិន រួចទើបបន្តរំកិលទៅកាន់កម្រិតបន្ទាប់។ ក្បួនដោះស្រាយនេះប្រើប្រាស់រចនាសម្ព័ន្ធ Queue (FIFO) និងមានល្បឿន O(V + E)។
• ករណីសិក្សា៖ ប្រដូចទៅនឹងការទម្លាក់ដុំថ្មចូលទៅក្នុងទឹក រលកនឹងសាយភាយចេញជារង្វង់ពីជិតទៅឆ្ងាយ។ BFS ត្រូវបានប្រើប្រាស់នៅក្នុង Google Maps ដើម្បីគណនាស្វែងរក **ផ្លូវខ្លីបំផុត (Shortest Path)** ទៅកាន់គោលដៅ។

២. DFS (Depth-First Search - ស្វែងរកតាមទម្រង់ស៊ីជម្រៅ)

DFS ធ្វើការរុករកដោយជ្រើសរើសផ្លូវមួយ រួចដើរតម្រង់ទៅមុខស៊ីជម្រៅរហូតដល់ផ្លូវទាល់ ទើបចាប់ផ្តើមដកថយត្រឡប់ក្រោយ **(Backtrack)** មករកផ្លូវបំបែកផ្សេងទៀត។ ក្បួនដោះស្រាយនេះប្រើប្រាស់ Stack (LIFO) ឬការហៅខ្លួនឯង (Recursion)។
• ករណីសិក្សា៖ ប្រដូចទៅនឹងការរាវរកផ្លូវចេញពីវង្កត់ (Maze)។ DFS ត្រូវបាន AI ប្រើប្រាស់ជាចាំបាច់ដើម្បីត្រួតពិនិត្យ និងស្វែងរកថាតើបណ្តាញតភ្ជាប់មានកកើតឡើងនូវ **រង្វិលជុំ (Cycle)** ឬអត់។

បញ្ជាក់៖ នៅក្នុងការសរសេរកូដស្វែងរកផ្លូវ (BFS/DFS) យើងត្រូវប្រើប្រាស់ List ឬ Set មួយឈ្មោះថា visited ជានិច្ច ដើម្បីកត់ត្រាចំណុចដែលបានដើរឆ្លងកាត់រួចហើយ ការពារកុំព្យូទ័ររត់វិលវល់ក្នុងរង្វិលជុំ (Infinite Loop) គាំងម៉ាស៊ីន។

៩. ការប្រៀបធៀបលក្ខណៈរវាង Hash Table និង Array
ក្នុងការអភិវឌ្ឍន៍កម្មវិធីកម្រិតអាជីព ការសម្រេចចិត្តជ្រើសរើសរវាង Hash Table (Dictionary) និង Array (List) គឺអាស្រ័យទៅលើទម្រង់ប្រតិបត្តិការដែលប្រព័ន្ធតម្រូវការ៖
លក្ខណៈបច្ចេកទេស និងទិន្នន័យនៃការដោះដូរ (Trade-offs)
ដំណោះស្រាយរបស់ Hash Tables
- ស្វែងរកលឿនបំផុត O(1)៖ សម្រាប់ការស្វែងរកទិន្នន័យដោយប្រើប្រាស់ឈ្មោះ Key (លោតទៅចំកន្លែងភ្លាមៗ)។
- មិនបានរក្សាលំដាប់ (Unordered)៖ ទិន្នន័យមិនត្រូវបានរៀបតាមលំដាប់លំដោយ ឬតាម Index ឡើយ។
- ស៊ី Memory ច្រើន៖ ដោយសារត្រូវចំណាយទំហំសម្រាប់ត្រៀមទុកជៀសវាងការបុកគ្នា (Collision)។
ដែនកំណត់របស់ Arrays
- ស្វែងរកយឺត O(n)៖ ប្រសិនបើត្រូវស្វែងរកដោយផ្អែកលើតម្លៃទិន្នន័យ ព្រោះត្រូវប្រើ Loop ឆែកមើលម្តងមួយ។
- រក្សាលំដាប់ (Ordered)៖ ទិន្នន័យត្រូវបានតម្រៀបតាមលំដាប់លំដោយ Index (0, 1, 2...) ងាយស្រួលរុករក។
- សន្សំសំចៃ Memory៖ ប្រើប្រាស់ទំហំ RAM ជាក់លាក់ទៅតាមទិន្នន័យពិតប្រាកដ។
វិធានមាសសម្រាប់ការជ្រើសរើស (Golden Rule)

• ជ្រើសរើស Hash Table (Dictionary)៖ នៅពេលដែលអ្នកត្រូវដំណើរការស្វែងរក (Search) ឬបន្ថែម (Insert) ទិន្នន័យរាប់លានដងជាប្រចាំដោយផ្អែកលើឈ្មោះកូនសោសម្គាល់ (Key-Value)។
• ជ្រើសរើស Array (List)៖ នៅពេលដែលទិន្នន័យរបស់អ្នកត្រូវការការតម្រៀបលំដាប់លំដោយ (Sorted/Ordered) ឬត្រូវកែប្រែដំណើរការទិន្នន័យវិលជុំ (Iteration) ពីដើមដល់ចប់បញ្ជីជាចម្បង។

• Graph: ជារចនាសម្ព័ន្ធបណ្តាញសេរី គ្មាន Root ឡើយ ធាតុនីមួយៗអាចតភ្ជាប់ខ្វាត់ខ្វែងពហុទិសដៅ និងអាចបង្កើតជារង្វិលជុំ (Cycle) បានយ៉ាងងាយស្រួល។
២. ហេតុអ្វីបានជា BFS សាកសមបំផុតសម្រាប់ការស្វែងរកផ្លូវខ្លីបំផុត (Shortest Path)?
ការបកស្រាយ៖ BFS ដំណើរការរុករកដោយ **ស្កែនមើលរាល់គ្រប់ចំណុចអ្នកជិតខាងជុំវិញខ្លួនមុន** (កម្រិតទី១) រួចទើបបន្តរំកិលទៅកម្រិតទី២ និងទី៣។ ដំណើរការបែបសាយភាយរង្វង់មូលនេះធានាថា រាល់ពេលដែល BFS រកឃើញចំណុចគោលដៅជាលើកដំបូង បន្ទាត់គន្លងផ្លូវនោះគឺជាខ្សែផ្លូវដែលមានចម្ងាយខ្លីបំផុតជាក់ស្តែង O(V + E)។
៣. តើ Collision នៅក្នុង Hash Table កើតឡើងដោយសារអ្វី? តើ Chaining ដោះស្រាយវាដូចម្តេច?
ការបកស្រាយ៖ Collision កើតឡើងនៅពេលដែល Key ពីរផ្សេងគ្នា ត្រូវបានគណនាដោយ Hash Function ទៅឃើញតម្លៃ Index ជាន់គ្នា (ចង់រត់ចូលថតតែមួយដដែល)។ យន្តការ **Chaining ( separate chaining )** ដោះស្រាយវាដោយបង្កើត Linked List មួយនៅក្នុងប្រអប់នីមួយៗ។ ប្រសិនបើមានទិន្នន័យជាន់គ្នា វានឹងតភ្ជាប់កន្ទុយ Linked List នោះបន្ថែម O(1) ដែលជួយរក្សាទុកទិន្នន័យទាំងពីរបានដោយជោគជ័យ។
៤. នៅក្នុង Python ហេតុអ្វីបានជាយើងមិនអាចប្រើប្រាស់ List ធម្មតា ធ្វើជាកូនសោ (Key) របស់ Dictionary បាន?
ការបកស្រាយ៖ ពីព្រោះកូនសោ Key របស់ Dictionary ត្រូវតែជាប្រភេទ **Immutable (មិនអាចកែប្រែតម្លៃបាន)** ដើម្បីធានាថាតម្លៃគណនា Hash របស់វាមានតម្លៃថេរជានិច្ច ងាយស្រួលរកឃើញទិន្នន័យចំទីតាំង RAM។ ចំណែកឯ List គឺជាប្រភេទ Mutable (អាចកែប្រែទិន្នន័យខាងក្នុងបាន) បើយើងកែតម្លៃ List នោះ វានឹងធ្វើឱ្យកុំព្យូទ័រគណនា Hash លែងត្រូវ នាំឱ្យមានបញ្ហាកូដកំហុស (Unhashable Type)។
