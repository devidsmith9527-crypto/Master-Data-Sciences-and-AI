១. សេចក្តីផ្តើមអំពី Dynamic Programming (DP)
Dynamic Programming (DP ឬវិធីសាស្ត្រដោះស្រាយកម្មវិធីបែបឌីណាមិក) គឺជាបច្ចេកទេសបង្កើនប្រសិទ្ធភាពកូដ (Optimization Technique) ដ៏មានឥទ្ធិពលបំផុតនៅក្នុងវិទ្យាសាស្ត្រកុំព្យូទ័រ។ គោលការណ៍គ្រឹះរបស់ DP គឺសាមញ្ញបំផុត៖ "ការចងចាំចម្លើយចាស់ ដើម្បីកុំឱ្យគណនាបញ្ហាដដែលៗឡើងវិញ (Solve subproblems once and store the results)"។ វាធ្វើការបំបែកបញ្ហាធំៗស្មុគស្មាញ ទៅជាបញ្ហាតូចៗតម្រួតលើគ្នា (Overlapping Subproblems) រួចរក្សាទុកលទ្ធផលគណនាតូចៗទាំងនោះនៅក្នុង Memory។
ករណីសិក្សា៖ របៀបដែលមនុស្សគិតគូរបូកលេខ (Human Intuition)

សាកល្បងសួរក្មេងម្នាក់ថា៖ តើផលបូក 1 + 1 + 1 + 1 + 1 ស្មើនឹងប៉ុន្មាន?
ក្មេងនោះនឹងចាប់ផ្តើមរាប់ថា "មួយ ពីរ បី បួន ប្រាំ" រួចឆ្លើយយ៉ាងលឿនថា៖ ស្មើ ៥។
ចុះប្រសិនបើយើងសរសេរថែម "+ 1" នៅខាងចុងសមីការនោះមួយទៀត តើក្មេងនោះត្រូវការចាប់ផ្តើមរាប់តាំងពី "មួយ ពីរ បី..." ឡើងវិញដែរឬទេ?
ចម្លើយពិតប្រាកដគឺ ទេ! ព្រោះខួរក្បាលរបស់គេបាន **"ចងចាំចម្លើយចាស់"** រួចស្រាប់ទៅហើយថាបណ្តុំមុនស្មើ ៥ ដូចនេះគេគ្រាន់តែយក ៥ បូកបន្ថែម ១ ស្មើ ៦ ភ្លាមៗ O(1)។ នេះហើយជាគោលការណ៍គ្រឹះរបស់ Dynamic Programming នៅក្នុងកុំព្យូទ័រ!

នៅក្នុងការសរសេរកូដធម្មតា (ដូចជាការគណនា Recursion ធម្មតា) កុំព្យូទ័រអត់មានខួរក្បាលចងចាំចម្លើយចាស់ឡើយ។ វានឹងរត់គណនាផលបូកចាស់ៗឡើងវិញរាប់លានដង ដែលជារឿយៗធ្វើឱ្យល្បឿនកូដធ្លាក់ចុះដល់កម្រិតយឺតបំផុតគឺស្វ័យគុណ O($2^n$)។ ការប្រើប្រាស់ DP ជួយបង្កើនល្បឿនឱ្យមកនៅត្រឹមលីនេអ៊ែរ O($n$) ដែលលឿនជាងមុនរាប់លានដង។

២. យន្តការ Memoization (Top-Down) ធៀបនឹង Tabulation (Bottom-Up)
នៅក្នុងការអនុវត្ត Dynamic Programming វិស្វករ DSA ប្រើប្រាស់វិធីសាស្ត្រធំៗពីរក្នុងការបង្កើតប្រព័ន្ធចងចាំ (Cache System) ទៅតាមទម្រង់ការសរសេរកូដ៖
ភាពខុសគ្នារវាងការសរសេរកូដបែប Top-Down និង Bottom-Up
១. Top-Down (Memoization)
- គោលការណ៍៖ ចាប់ផ្តើមគិតពីបញ្ហាធំបង្អស់ រួចប្រើប្រាស់យន្តការ Recursion ដើម្បីបំបែកបញ្ហានោះទៅជាបញ្ហាតូចៗបំផុត (Base Cases)។
- យន្តការចងចាំ៖ រាល់ពេលដែលរកឃើញចម្លើយតូចណាមួយ ប្រព័ន្ធនឹងយកចម្លើយនោះទៅ **កត់ត្រាទុកក្នុង Dictionary ឬ Hash Table** (ហៅថា `memo`) មុននឹង Return។ ពេលជួបបញ្ហានោះម្តងទៀត វានឹងទាញយកចម្លើយមកប្រើភ្លាម O(1)។
២. Bottom-Up (Tabulation)
- គោលការណ៍៖ ចាប់ផ្តើមដោះស្រាយពីបញ្ហាតូចបំផុតជាមុន រួចបន្តសន្សំចម្លើយឡើងលើរហូតដល់សម្រេចបានបញ្ហាធំ (លែងប្រើ Recursion)។
- យន្តការចងចាំ៖ ប្រព័ន្ធប្រើប្រាស់ **តារាង Array ឬ List** (ហៅថា `table`) ដើម្បីរៀបចំគណនាទិន្នន័យពីលំដាប់តូចទៅធំដោយប្រើប្រាស់ Loop `for` ធម្មតា។ វិធីសាស្ត្រនេះមានស្ថេរភាពខ្ពស់ ព្រោះមិនបារម្ភបញ្ហា Stack Overflow (Recursion Limit) ឡើយ។

៣. ការអនុវត្តកូដ DP លើលំដាប់លេខ Fibonacci ក្នុង Python
លំដាប់លេខ Fibonacci (0, 1, 1, 2, 3, 5, 8...) គឺជាករណីសិក្សាល្អបំផុតក្នុងការបង្ហាញពីអនុភាពរបស់ Dynamic Programming។ ខាងក្រោមនេះជាកូដប្រៀបធៀបរវាងការសរសេរ Recursion ធម្មតា និងការប្រើប្រាស់ DP (Memoization)៖
1_fibonacci_dp.pyRecursive vs DP Fibonacci
import time

# ១. វិធីសាស្ត្រ Recursion ធម្មតា (អត់មានការចងចាំ - យឺតខ្លាំង O(2ⁿ))
def fib_recursive(n):
    if n <= 1:
        return n
    return fib_recursive(n-1) + fib_recursive(n-2)

# ២. វិធីសាស្ត្រ Dynamic Programming (Memoization - លឿន O(n))
def fib_dp(n, memo={}):
    # បើមានចម្លើយចាស់ក្នុង Dictionary រួចហើយ ទាញយកមកប្រើភ្លាម O(1)
    if n in memo:
        return memo[n]
    if n <= 1:
        return n
    
    # គណនា រួចកត់ត្រាទុកក្នុង Dictionary memo
    memo[n] = fib_dp(n-1, memo) + fib_dp(n-2, memo)
    return memo[n]

# វាស់ស្ទង់ល្បឿនជាក់ស្តែង
print("--- ប្រៀបធៀបល្បឿនគណនា Fibonacci ទី ៣៥ ---")

start = time.time()
ans1 = fib_recursive(35)
print(f"Recursion ធម្មតា គណនាឃើញ {ans1} ប្រើពេល: {time.time() - start:.4f} វិនាទី")

start = time.time()
ans2 = fib_dp(35)
print(f"DP (Memoization) គណនាឃើញ {ans2} ប្រើពេល: {time.time() - start:.4f} វិនាទី")

- ភាគរយត្រឹមត្រូវ៖ មិនធានាថាចម្លើយដែលទទួលបានគឺល្អបំផុតជាសកល (Global Optimum) នោះឡើយ ជារឿយៗតែងតែចាញ់បោកក្នុងបញ្ហា Knapsack ព្រោះវាប្រញាប់យករបស់ធំថ្លៃមុនគេ ធ្វើឱ្យកាតាបពេញគ្មានចន្លោះ។
២. Dynamic Programming (ចម្លើយជាធំ)
- យន្តការសម្រេចចិត្ត៖ ពិចារណារាល់គ្រប់លទ្ធភាពជម្រើសតូចៗទាំងអស់ (Overlapping Subproblems)។
- ល្បឿន (Time Complexity)៖ យឺតជាង Greedy កម្រិត O($n \times W$) និងប្រើប្រាស់ទំហំ RAM ច្រើនជាងដើម្បីសង់តារាង Tabulation។
- ភាគរយត្រឹមត្រូវ៖ ធានាភាពត្រឹមត្រូវដាច់ខាត ១០០% ថាចម្លើយសម្រេចចិត្តដែលទទួលបានគឺល្អបំផុតជាសកល (Global Optimum) គ្មានចម្លើយណាអាចល្អជាងនេះឡើយ។
ករណីចាញ់បោករបស់ Greedy លើ Coin Change Problem

ឧបមាថាយើងត្រូវអាប់លុយចំនួន **៦,០០០៛** ដូចមុន។ ប៉ុន្តែក្នុងថតលុយមានតែក្រដាសប្រាក់៖ **៤,០០០៛, ៣,០០០៛, និង ១,០០០៛**។
• ប្រសិនបើប្រើប្រាស់ក្បួន Greedy៖ វានឹងរើសក្រដាសធំមុនគេគឺ **៤,០០០៛** (សល់ ២,០០០៛) រួចបន្តអាប់ក្រដាស **១,០០០៛** ចំនួន ២ សន្លឹក។ សរុបប្រើប្រាស់ក្រដាសប្រាក់អស់ **៣ សន្លឹក**។
• ប្រសិនបើប្រើប្រាស់ក្បួន Dynamic Programming៖ វានឹងគណនាឃើញជម្រើសល្អបំផុតគឺការអាប់ក្រដាស **៣,០០០៛** ចំនួន **២ សន្លឹក**។ សរុបប្រើប្រាស់ក្រដាសប្រាក់អស់តែ **២ សន្លឹក** ប៉ុណ្ណោះ។ នេះសរបញ្ជាក់ថា Greedy មិនអាចផ្តល់ចម្លើយល្អបំផុតគ្រប់ពេលឡើយ។

៨. ក្បួនដោះស្រាយរបស់ Dijkstra (Dijkstra's Shortest Path)
Dijkstra's Algorithm (ក្បួនដោះស្រាយរបស់ឌីកស្ត្រា) គឺជាក្បួនដោះស្រាយដ៏ល្បីល្បាញបំផុតនៅក្នុងទ្រឹស្តីក្រាហ្វ (Graph Theory) សម្រាប់ស្វែងរក ផ្លូវខ្លីបំផុត (Shortest Path) ពីចំណុចចាប់ផ្តើម (Source Vertex) ទៅកាន់ចំណុចគោលដៅផ្សេងទៀតទាំងអស់ នៅក្នុងក្រាហ្វដែលមានទម្ងន់ខុសៗគ្នា (Weighted Graph)។
ការយល់ដឹងពី "Cost / Weight" (ទម្ងន់ខ្សែ)

នៅក្នុងការស្វែងរកផ្លូវពិតប្រាកដ ពាក្យថា "ផ្លូវខ្លីបំផុត" មិនមែនគិតទៅលើចម្ងាយផ្លូវ (គីឡូម៉ែត្រ) តែម្ខាងនោះឡើយ។ តម្លៃទម្ងន់ (Cost) អាចតំណាងឱ្យ៖
• រយៈពេលធ្វើដំណើរ៖ ផ្លូវខ្លីតែបើជួបការស្ទះចរាចរណ៍ខ្លាំង នោះ Cost នឹងកើនឡើងខ្ពស់ខ្លាំង។
• កម្រិតស៊ីប្រេង ឬថ្លៃផ្លូវល្បឿនលឿន៖ ផ្លូវវាងអាចឆ្ងាយ បើឡានរត់លឿនមិនសូវស៊ីសាំង នោះកុំព្យូទ័រនឹងជ្រើសរើសផ្លូវវាងនោះ។

ករណីសិក្សានៃប្រព័ន្ធផែនទី Google Maps

រាល់ពេលដែលអ្នកវាយបញ្ចូលទីតាំងធ្វើដំណើរពី "ភ្នំពេញ ទៅ សៀមរាប" នៅក្នុង Google Maps ប្រព័ន្ធបញ្ញាសិប្បនិម្មិតនឹងដំណើរការក្បួនកូនកាត់របស់ Dijkstra (ហៅថា A* Search) ដើម្បីវិភាគទិន្នន័យចរាចរណ៍រស់ (Real-time traffic) រួចផ្តល់នូវជម្រើសខ្សែផ្លូវដែលចំណាយពេលតិចបំផុតជូនអ្នក O(E + V $\log$ V)។

៩. ក្បួនដោះស្រាយ K-Nearest Neighbors (KNN) សម្រាប់ប្រព័ន្ធ AI
K-Nearest Neighbors (KNN ឬក្បួនស្វែងរកអ្នកជិតខាងជិតបំផុតចំនួន K នាក់) គឺជាក្បួនដោះស្រាយការរៀនសូត្ររបស់ម៉ាស៊ីន (Machine Learning) បែបមូលដ្ឋាន និងងាយស្រួលយល់បំផុត។ វាត្រូវបានចាត់ទុកថាជា **"Lazy Learner"** ព្រោះវាមិនមានការហ្វឹកហាត់បង្កើតជាគំរូស្មុគស្មាញ (Model-free) ឡើយ គឺវាគ្រាន់តែរក្សារក្សាទុកទិន្នន័យចាស់ៗទាំងអស់នៅក្នុង RAM។

ទស្សនៈគ្រឹះរបស់ KNN៖ "You are who your friends are"

ដើម្បីកំណត់អត្តសញ្ញាណទិន្នន័យថ្មីមួយ (ឧ. ចង់ដឹងថាអតិថិជនថ្មីចូលចិត្តមើលកុនបែបណា?) KNN នឹងគណនាចម្ងាយគម្លាត **(Euclidean Distance - ចម្ងាយវាស់ដោយខ្សែបន្ទាត់ត្រង់)** ពីអតិថិជនថ្មីនោះ ទៅកាន់អតិថិជនចាស់ៗទាំងអស់។ រួចទាញយកអតិថិជនចាស់ៗដែលនៅកៀកក្បែរគាត់បំផុតចំនួន **K នាក់** (ឧ. K = 5 នាក់) រួចធ្វើការសម្រេចចិត្តវិនិច្ឆ័យទៅតាម **មតិភាគច្រើន (Majority Vote)** របស់អតិថិជនទាំង ៥ នាក់នោះ។

ទោះបីជា KNN ងាយស្រួលសរសេរកូដ និងមានភាគរយភាពត្រឹមត្រូវខ្ពស់ក៏ដោយ ក៏វាមានបញ្ហាដ៏ធំមួយគឺ **ល្បឿនយឺត O(n × d) និងស៊ី Memory ខ្លាំង O(n)** ពេលទិន្នន័យចាស់ៗកើនឡើងដល់រាប់លាន។ មូលហេតុគឺដោយសារតែរាល់ពេលដែលមានទិន្នន័យថ្មីមួយចូលមក កុំព្យូទ័រត្រូវបង្ខំចិត្តគណនា Euclid Distance ទៅកាន់គ្រប់ទិន្នន័យចាស់ៗទាំងអស់ម្តងមួយៗ O(n)។ ដើម្បីដោះស្រាយបញ្ហានេះ នៅក្នុងប្រព័ន្ធ AI ខ្នាតយក្ស គេប្រើប្រាស់ក្បួនចម្លង **Approximate Nearest Neighbor (ANN)** ជំនួសវិញ ដើម្បីបង្កើនល្បឿនឲ្យលឿន O($\log n$)។

១១. សំណួររំលឹកមេរៀនប្រចាំជំពូកទី ៨ និងការបកស្រាយលម្អិត (Q&A)
ខាងក្រោមនេះគឺជាសំណួរ និងការបកស្រាយគំរូដ៏សំខាន់ៗដែលត្រូវបានរៀបចំឡើងដើម្បីវាស់ស្ទង់ការយល់ដឹងរបស់និស្សិតថ្នាក់អនុបណ្ឌិត៖
១. តើអ្វីជាភាពខុសគ្នាជាមូលដ្ឋានរវាងយន្តការ Memoization និង Tabulation នៅក្នុង DP?
• Memoization (Top-Down): ចាប់ផ្តើមពីបញ្ហាធំ បំបែកទៅបញ្ហាតូចៗដោយប្រើប្រាស់ Recursion និងកត់ត្រាចម្លើយចាស់ទុកនៅក្នុង Dictionary (ស៊ី Memory RAM ច្រើន និងអាចជួបបញ្ហា Stack Overflow)។
• Tabulation (Bottom-Up): ចាប់ផ្តើមដោះស្រាយពីបញ្ហាតូចៗឡើងលើដោយប្រើប្រាស់ Loop `for` ធម្មតា និងកត់ត្រាចម្លើយទុកនៅក្នុងតារាង Array (មានលំនឹងល្អ និងលឿនជាង Memoization)។
២. ហេតុអ្វីបានជា Greedy Algorithm ជារឿយៗតែងតែបរាជ័យក្នុងការដោះស្រាយ 0/1 Knapsack Problem?
ការបកស្រាយ៖ ដោយសារតែ Greedy សម្រេចចិត្តលោភលន់ចំពោះមុខ **(Local Optimum)** ដោយរើសយករបស់ណាដែលមានតម្លៃថ្លៃបំផុតដាក់កាតាបជាមុនជានិច្ច។ ការធ្វើបែបនេះអាចបណ្តាលឱ្យកាតាបពេញទម្ងន់ភ្លាមៗ ធ្វើឱ្យយើងបាត់បង់ឱកាសក្នុងការដាក់របស់តូចៗជាច្រើនមុខផ្សេងទៀត ដែលបើផ្គុំបញ្ជូលគ្នាទៅមានតម្លៃចំណេញលុយខ្ពស់ជាងរបស់ថ្លៃតែមួយនោះឆ្ងាយណាស់។ ដូចនេះមានតែការគណនាគ្រប់ជម្រើស Subproblems របស់ Dynamic Programming ទេ ទើបធានាបាននូវភាពត្រឹមត្រូវ Global Optimum ១០០%។
៣. នៅក្នុង Dijkstra's Algorithm តើ "ទម្ងន់ខ្សែ (Cost / Weight)" ដើរតួនាទីយ៉ាងដូចម្តេចពិតប្រាកដ?
ការបកស្រាយ៖ Cost ដើរតួនាទីជា **តម្លៃចំណាយ ឬរយៈពេលរារាំង**។ នៅក្នុងការស្វែងរកផ្លូវពិតប្រាកដ ផ្លូវដែលខ្លីបំផុតមិនមែនគិតទៅលើចម្ងាយ (គីឡូម៉ែត្រ) តែម្ខាងនោះឡើយ។ តម្លៃ Cost អាចតំណាងឱ្យកម្រិតស្ទះចរាចរណ៍ ឬតម្លៃសាំង។ ដូចនេះ Dijkstra's Algorithm នឹងគណនារកខ្សែផ្លូវណាដែលមានតម្លៃរួមបញ្ចូល Cost តិចបំផុត O(E + V $\log$ V) ជួយឱ្យការធ្វើដំណើរកាន់តែលឿន ទោះជាត្រូវជិះផ្លូវវាងឆ្ងាយជាងក៏ដោយ។
៤. តើអ្វីទៅជាឥទ្ធិពលរបស់ "K" នៅក្នុងក្បួនដោះស្រាយ K-Nearest Neighbors (KNN) របស់ប្រព័ន្ធ AI?
ការបកស្រាយ៖ Value "K" គឺជា **ចំនួនអ្នកជិតខាងជិតបំផុតដែលត្រូវយកមកបោះឆ្នោតសម្រេចចិត្ត**៖
• ប្រសិនបើ K តូចពេក (ឧ. K = 1)៖ ប្រព័ន្ធនឹងងាយរងគ្រោះថ្នាក់ដោយសារទិន្នន័យលម្អៀង ឬទិន្នន័យខូចខាត (Overfitting / High Variance)។
• ប្រសិនបើ K ធំពេក (ឧ. K = 100)៖ ប្រព័ន្ធនឹងលែងសូវមានភាពត្រឹមត្រូវ ព្រោះវាទាញយកទិន្នន័យពីចម្ងាយឆ្ងាយដែលមិនពាក់ព័ន្ធ មកបោះឆ្នោតឡូឡាបញ្ជូលគ្នាបង្ខំយកតាមមតិភាគច្រើន (Underfitting / High Bias)។ ដូចនេះ វិស្វករ AI តែងតែរើសយកលេខសេសតូចល្មម (ឧ. K = 3, 5, 7) ដើម្បីរក្សាតុល្យភាព។
